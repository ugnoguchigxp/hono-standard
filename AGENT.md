# 実装ルール (Agents Rules)

## 1. モジュラー・モノリス構成
- **バックエンド (`api/modules/<domain>/`)**
  - `*.routes.ts`: ルーティングとリクエスト/レスポンスハンドリングのみ（ビジネスロジックは書かない）。
  - `*.service.ts`: ビジネスロジック。
  - `*.repository.ts`: データベース(Drizzle)へのアクセス(CRUD)。
- **フロントエンド (`src/modules/<domain>/`)**
  - `components/`: ドメイン固有UI。
  - `hooks/`: 状態管理等のカスタムフック。
  - `repositories/`: APIデータフェッチ層 (TanStack Query)。
  - `services/`: フロントエンド固有のロジック。
- **共有スキーマ (`shared/schemas/`)**
  - Zodを用いたバリデーションを配置し、フロントとバックで型を完全共有する。

## 2. API 通信
- 生の `fetch` 利用は禁止。
- 必ず `src/lib/api.ts` の Hono RPC APIクライアント (`client`) を使用する（トークン付与やリフレッシュが内包されているため）。

## 3. 基本技術ルール
- **ルーティング**: TanStack Router (`src/routes/`)
- **状態管理・データ取得**: TanStack Query
- **DB操作**: Drizzle ORM
- **UIコンポーネント・スタイリング**:
  - **Tailwind CSS + shadcn/ui** をベースとしたセマンティックな設計。
  - デザイントークンは CSS変数 (`:root`, `.dark`) を用いて一元管理する（ライトテーマ/ダークテーマ対応）。
  - AIデザインツール **Pencil (pencil.dev)** 等を活用し、UI/UX 統合（Design-to-Code）を以下のルールで推進する。
    - **ルール1**: コンポーネントを新規作成した際は、必ず Pencil にコンポーネント一覧(パレット等)として追加し、デザイン上の共有資産とすること。
    - **ルール2**: フロントエンドの各モジュール (`src/modules/<domain>`) には、そのモジュール内のコンポーネントを包括してマウントしたプレビュー用コンポーネントを作成し、それと紐づく Pencil ファイル (`.pen`) を配置すること。これにより、ブラウザに頼らずカジュアルにモジュールの見た目を確認できるようにする。

## 4. ソフトウェア設計の基本原則
- **KISS**: 極力シンプルに。トリッキーな実装は避ける。
- **YAGNI**: 推測に基づく過度な汎用化や、先回りした抽象化は実装しない。
- **DRY**: ロジックの重複は避けるが、偶然の一致によるむやみな共通化（結合度増大）は避ける。
- **SRP (単一責任の原則)**: 関数・モジュールは「変更する理由が一つ」になる設計にする。

## 5. ベストプラクティス・心得
- **可読性最優先**: 「書く」より「読む」時間が長いことを前提に、妥協のない明確な命名（変数・関数）を行う。
- **早期リターン (Early Return)**: ガード節でエラー・例外を早めに `return` し、ネストを浅く保つ。
- **堅牢なエラーハンドリング**: 異常時は例外を握り潰さず、エラーを適切にハンドリング・伝播させる。
- **ボーイスカウトルール**: 改修等で触れたコードの周辺の小さなリファクタリングを継続的に行う。
- **オーバーエンジニアリング回避**: 問題のサイズに見合わない無駄な抽象化やデザインパターン適用はしない。

## 6. テストのルール
- **脆いテスト (Flaky Tests) の排除**: 現在日時・乱数・実行順への依存を作らない。
- **UIスナップショットテストの禁止**: HTML/CSSクラスの丸ごと保存はせず、ユーザーの「振る舞いと状態」を検証する。
- **実装の詳細をテストしない**: 「内部変数がどう変わったか」ではなく「この入力に対し正しい出力が得られるか (ブラックボックステスト)」を書く。
- **カバレッジを自己目的にしない**: カバレッジ稼ぎの空虚なテストは量産しない。コアロジックを優先する。

## 7. デザインシステム / CSS 変更のルール
- **CSS変数の間接参照を理解してから変更すること**: Tailwind の `rounded-sm`, `shadow-md` 等のユーティリティクラスは `tailwind.config.ts` で CSS変数（`var(--radius)` 等）に紐付いている場合がある。クラス名を変えても見た目が変わらないときは、まず `tailwind.config.ts` の定義を確認すること。固有のピクセル値が必要な場合は `rounded-[6px]` のように Tailwind の任意値構文を使う。
- **変更後は必ず Storybook で目視確認すること**: コード変更だけで「反映されたはず」と報告してはならない。Storybook を起動し、該当コンポーネントの**全バリアント**をブラウザで確認した上で完了報告する。

## 8. 個人情報保護とログ
- **個人情報・認証情報をログに残さないことを徹底する**。個人情報保護法の観点から、次をログ出力禁止とする。
  - メールアドレス、電話番号、住所、氏名などの個人情報
  - OAuth の `code` / `state`、アクセストークン、リフレッシュトークン、JWT、Cookie値、Authorization ヘッダー
  - そのほか個人を識別しうる情報
- **URLは原則として `path` のみを記録し、`query` は記録しない**。
- デバッグ目的でも機微情報の生値は出力しない。必要な場合はマスク済み値のみ扱う。
