# Phase 2: Flutter 入力機能 実装計画

## 概要

Flutter モバイルアプリを同一 monorepo に追加し、健康データの入力・同期・閲覧を成立させるフェーズ。
このフェーズでは、`flutter-template` に近い feature-first 構成を採用しつつ、ヘルスレコード用途に必要な機能だけを実装する。

対象は次の 4 領域に絞る。

- 血圧の手入力
- 血糖値の手入力
- 食事の品目と概算カロリー入力
- 運動情報の同期と閲覧

病院連携、医療者向け管理画面、Google Fit 直結実装は v1 では扱わない。

## 前提条件

- Phase 1 で健康記録 API と共通スキーマが用意されていること
- `mobile/` の Flutter プロジェクト雛形が存在する、またはこのフェーズの冒頭で作成すること
- 既存バックエンドは cookie ベースの認証を使っていること
- Flutter 側は `health` パッケージで iOS は HealthKit、Android は Health Connect を対象にすること
- `--dart-define=API_BASE_URL=...` で接続先を切り替えること
- 健康記録 API は現状 `/api/v1/health` 配下で提供されていること

## 実装方針

### Flutter 構成

`flutter-template` の構成思想をそのまま流用する。

- `lib/main.dart`
- `lib/bootstrap.dart`
- `lib/src/app/`
- `lib/src/core/`
- `lib/src/features/`
- `assets/locales/`
- `mockServer/`
- `integration_test/`

アプリ全体は feature-first にし、各機能は `presentation` と `data` を近接配置する。

### 依存関係

基本パッケージは以下を前提にする。

- `flutter_hooks`
- `hooks_riverpod`
- `flutter_riverpod`
- `go_router`
- `dio`
- `dio_smart_retry`
- `freezed_annotation`
- `json_annotation`
- `logger`
- `intl`
- `graphic`

追加で必要になるもの。

- `health` - 端末健康データ連携
- cookie 永続化用の HTTP クライアント補助
- オフライン下書き・同期キュー用のローカル保存層

### 認証方針

- Web と同じ認証基盤を使う
- Flutter 側は `flutter_secure_storage` にアクセストークンを保存する前提にしない
- cookie の永続化でセッションを維持する
- 401 を受けたら refresh を試行し、失敗時はログイン画面へ戻す

### UI/画面方針

- モバイルは記録入力の導線を最短にする
- Web は閲覧中心、Flutter は入力中心にする
- 画面遷移は `go_router` で統一する
- 日次サマリ、履歴、設定は共通のルート設計にする

## リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| HealthKit / Health Connect の権限取得が不安定 | 連携機能の利用率低下 | 手入力を常に残し、権限拒否を通常分岐にする |
| cookie セッションの扱いが不安定 | ログイン維持の失敗 | Dio に cookie 永続化を入れ、refresh と logout を明示化する |
| オフライン同期の重複 | データ不整合 | 外部 ID と送信状態で重複排除する |
| Flutter 雛形が未整備 | 開発開始の遅れ | Week 0 でテンプレート準拠の `mobile/` を先に作る |
